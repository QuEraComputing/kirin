# U4: Parser/Printer Codegen -- Ergonomics/DX Review

## Toy Scenario

I am defining a quantum gate dialect. I want to write a CNOT gate:

```rust
#[derive(Dialect, HasParser, PrettyPrint)]
#[kirin(type = QubitType)]
#[chumsky(format = "$cnot {ctrl}, {tgt} -> {ctrl_out:type}, {tgt_out:type}")]
pub struct Cnot {
    ctrl: SSAValue,
    tgt: SSAValue,
    ctrl_out: ResultValue,
    tgt_out: ResultValue,
}
```

This parses: `%c_out, %t_out = cnot %c, %t -> Qubit, Qubit`

The result names (`%c_out, %t_out =`) are handled generically by the framework. My format string only specifies the operation body and result types. This is clean.

Edge case: What if I forget `{tgt}` in the format string? The validation gives: "field 'tgt' is not mentioned in the format string. All fields must appear unless they have a default value." Clear and actionable.

Edge case: A graph-body operation with partial projections:

```rust
#[chumsky(format = "circuit ({body:ports}) {{ {body:body} }}")]
pub struct Circuit {
    body: DiGraph,
}
```

Validation error: "DiGraph field 'body' uses body projections but is missing required projection(s): :captures." This catches the mistake at compile time.

## Findings

### [P2] [likely] Format string syntax is not documented outside source code -- format.rs

The format string mini-language (`$keyword`, `{field}`, `{field:type}`, `{field:ports}`, `{:name}`, `{{` for escaping, `{sig:inputs}`, `{sig:return}`) is powerful but has no user-facing reference. Users must read format.rs tests or existing dialect code to learn the syntax. A table in the crate-level doc would significantly reduce onboarding time.

### [P2] [confirmed] `$keyword` vs `{.keyword}` migration error could be more helpful -- format.rs:184-196

The legacy `{.keyword}` check is good, but only triggers on `{.` -- users who never used the old syntax still need to discover `$keyword` exists. The first format string they write will likely be `{.add}` (by analogy to Python) or just `add` (literal). A hint in the format parse error like "Did you mean `$add`?" for unrecognized identifiers would help.

### [P3] [uncertain] Body projection completeness checking is strict -- validation.rs:128-175

Requiring all projections (`:ports`, `:captures`, `:body` for DiGraph) is correct for roundtrip, but forces verbose format strings even when captures are unused. A `#[kirin(no_captures)]` escape hatch for graph types that never use captures would reduce noise.

### [P1] [confirmed] RenderDispatch derive is minimal but undocumented -- derive-prettyless/generate.rs

`#[derive(RenderDispatch)]` generates a simple match-arm dispatch for stage enums, but there is no doc comment on the derive macro explaining what it does, when to use it, or what traits the inner types must implement. Users discover it only by reading the toy-lang example.

## Concept Budget Table

| Concept | Where learned | Complexity |
|---------|--------------|------------|
| Format string syntax | format.rs tests / dialect examples | Med |
| `$keyword` namespaced keywords | format.rs | Low |
| `{field:type}` / `{field:name}` projections | format.rs | Low |
| `{field:ports}` / `{field:body}` body projections | format.rs | Med |
| `{sig:inputs}` / `{sig:return}` signature projections | format.rs | Med |
| `{:name}` context projection | format.rs | Med |
| `#[chumsky(format = "...")]` attribute | dialect examples | Low |
| Validation rules (completeness, result name rejection) | validation.rs | High |

## Lifetime Complexity

(i) **Hidden by derive**: All parser/printer lifetimes generated by `#[derive(HasParser, PrettyPrint)]` -- users never write them.
(ii) **Visible necessary**: None for dialect authors.
(iii) **Visible avoidable**: None.

## Strengths

- Format string is genuinely intuitive for the common case: `"$add {lhs}, {rhs} -> {res:type}"` reads like a specification.
- Compile-time validation catches missing fields, invalid projections, and legacy syntax -- no runtime surprises.
- Auto-placeholder for ResultValue means simple ops need only `{res:type}` or nothing at all.
- Signature projections (`{sig:inputs}`, `{sig:return}`) decouple signature layout from body layout elegantly.
- Legacy syntax migration error is a nice DX touch.
