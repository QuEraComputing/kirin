# U5: Output & Dialects -- Ergonomics/DX Review

## Toy Scenario

I have composed a language from kirin-function's `Lexical` + my custom `TweezerOps` dialect. I want to print the whole pipeline.

```rust
use kirin_prettyless::prelude::*;

// Single statement
let output = my_statement.sprint(&stage);

// Whole pipeline
let output = pipeline.sprint();

// With config
let output = pipeline.render()
    .config(Config::new().with_width(80))
    .into_string()?;

// A single function across stages
let output = my_func.sprint(&pipeline);
```

This is clean. The `sprint` shorthand covers 90% of cases. The builder pattern covers the rest. No lifetime gymnastics needed.

Now, composing a language enum:

```rust
#[derive(Dialect, HasParser, PrettyPrint, Interpretable, SSACFGRegion)]
#[kirin(builders, type = ArithType)]
pub enum MyLang {
    #[wraps] #[callable]
    Lexical(Lexical<ArithType>),
    #[wraps]
    TweezerOps(TweezerOps<ArithType>),
}
```

Five derives on one line. This is the cost of the "all three required" dialect contract. Each derive is essential, but the line is visually dense.

## Findings

### [P2] [confirmed] Five derives are required for every language enum -- kirin-function/lib.rs:36

`Dialect, HasParser, PrettyPrint, Interpretable, SSACFGRegion` -- the toy-lang example has 6 derives per enum. For a physicist who just wants to compose existing dialects, this is intimidating. A `#[derive(Language)]` meta-derive that expands to the standard set would reduce the visual burden from `Dialect, HasParser, PrettyPrint, Interpretable, SSACFGRegion` to `Language`.

### [P2] [likely] PrettyPrint trait requires `L::Type: Display` at every method -- traits.rs:41-54

Every `PrettyPrint` method has a `where L::Type: std::fmt::Display` clause. Dialect authors implementing manual `PrettyPrint` must copy-paste this bound. Since `CompileTimeValue` already requires `Display`, this bound is always satisfied -- it could be baked into the `Dialect` supertrait or into `PrettyPrint`'s own supertrait bound.

### [P1] [confirmed] `PrettyPrint` and `PrettyPrintExt` naming is confusing -- traits.rs

`PrettyPrint` is the impl trait (you implement it). `PrettyPrintExt` is the call-site trait (you call `.sprint()` / `.render()`). But from a user perspective, `.sprint()` is the primary interaction. Having to know about `PrettyPrintExt` to discover `sprint` and `render` adds one concept that could be on `PrettyPrint` itself via a provided method. The blanket impl approach is correct for separation of concerns, but the naming does not communicate the relationship.

### [P3] [uncertain] `kirin-function` Lexical vs Lifted distinction requires compiler knowledge -- lib.rs:1-16

The doc comment explains the difference well, but a physicist will ask: "which one do I use?" The answer depends on whether your language has closures (Lexical) or top-level functions only (Lifted). A one-line recommendation in the doc would help: "Start with `Lexical` unless you are writing a lowering pass."

## Concept Budget Table

| Concept | Where learned | Complexity |
|---------|--------------|------------|
| PrettyPrint / PrettyPrintExt | traits.rs | Low |
| sprint / render builder | traits.rs | Low |
| RenderDispatch | pipeline.rs | Med |
| PipelinePrintExt / PrintExt | pipeline.rs | Med |
| Config (width, tab_spaces) | config.rs | Low |
| Document (manual printing) | document/builder.rs | High |
| Lexical vs Lifted | kirin-function/lib.rs | Med |
| #[wraps] + #[callable] composition | example/toy-lang | Med |

## Lifetime Complexity

(i) **Hidden by derive**: `PrettyPrint` impls generated by `#[derive(PrettyPrint)]` -- users never write the `<'a, L>` generics.
(ii) **Visible necessary**: `Document<'a, L>` lifetime when writing manual `PrettyPrint` impls -- needed to tie arena allocations to the document.
(iii) **Visible avoidable**: The `where L::Type: std::fmt::Display` clause on every `PrettyPrint` method could be moved to a supertrait bound, removing it from manual impls.

## Strengths

- `sprint` as a one-method shorthand is exactly right for exploration workflows.
- The builder pattern (`render().config(...).into_string()`) is clean and discoverable.
- `PrettyPrintViaDisplay` marker trait is a nice zero-cost way to get printing for simple types.
- `kirin-function`'s `Lexical` and `Lifted` enums demonstrate excellent composition: same `Call` and `Return` reused across both styles.
- The `bat` feature flag for pager output is a thoughtful DX touch for interactive development.
- Pipeline rendering (`pipeline.sprint()`) just works across heterogeneous stages thanks to `RenderDispatch`.
