# Physicist -- Ergonomics/DX Review: kirin-ir

## Repetition & Boilerplate

### 1. PhantomData ceremony on every generic dialect
Every generic dialect struct/enum must include a `__Phantom(PhantomData<T>)` variant (enums) or `#[kirin(default)] marker: PhantomData<T>` field (structs). This is required for the type parameter to be used, but it is pure ceremony from the user's perspective.

Examples from the codebase:
- `kirin-arith/src/lib.rs` line 128: `__Phantom(std::marker::PhantomData<T>)`
- `kirin-cf/src/lib.rs` line 47: `__Phantom(std::marker::PhantomData<T>)`
- `kirin-scf/src/lib.rs` lines 64, 82, 93: `#[kirin(default)] marker: std::marker::PhantomData<T>`

A user creating a tweezer control dialect with 5 struct operations would write `#[kirin(default)] marker: PhantomData<T>` five times. This could be auto-generated by the derive macro when `T` appears in `#[kirin(type = T)]` but not in any field.

### 2. Derive list repetition
Every dialect type requires the same long derive list:
```rust
#[derive(Debug, Clone, PartialEq, Eq, Hash, Dialect, HasParser, PrettyPrint)]
```
This is 8 derives. With interpreter support it grows to 10:
```rust
#[derive(Debug, Clone, PartialEq, Eq, Hash, Dialect, HasParser, PrettyPrint, Interpretable, SSACFGRegion)]
```

The standard Rust derives (`Debug, Clone, PartialEq, Eq, Hash`) are required by `Dialect`'s supertraits. Consider having `#[derive(Dialect)]` auto-derive those, or providing a shorthand macro.

### 3. `#[wraps]` on every variant
In the toy-lang `HighLevel` enum, every single variant has `#[wraps]`. When composing existing dialects (the normal use case), `#[wraps]` is always present. The non-wrapping case (raw struct fields) is the exception. Consider making `#[wraps]` the default for enum variants when the enum-level `#[wraps]` attribute is present.

Wait -- the enum-level `#[wraps]` already exists (see `kirin-function/src/lib.rs` line 38). The toy-lang example at `example/toy-lang/src/language.rs` uses per-variant `#[wraps]` instead. Both patterns work; the per-variant version requires more typing.

### 4. `#[kirin(builders, type = T)]` on every type
Every dialect type repeats `#[kirin(type = T)]`. This is the type lattice parameter and is fundamental, so the repetition is somewhat justified. But `builders` is also repeated on every type. Consider defaulting `builders` to on.

## Lifetime Complexity

### Dialect trait has HRTB in its supertraits
The `Dialect` trait definition (kirin-ir/src/language.rs:103-128) has 14 `for<'a>` bounds:
```rust
pub trait Dialect:
    for<'a> HasArguments<'a>
    + for<'a> HasResults<'a>
    + for<'a> HasArgumentsMut<'a>
    ...
```
Users never see this directly because `#[derive(Dialect)]` handles it. But if a user tries to write a manual `Dialect` impl (perhaps for a dynamically-typed dialect), the HRTB storm is daunting. This is acceptable since derive is the happy path, but documentation should strongly steer users toward derive.

### User-facing lifetimes are minimal
For the common workflow (define dialect -> compose -> parse -> interpret), users encounter:
- No explicit lifetime annotations on dialect definitions
- `'ir` on interpreter interactions, but only when writing manual `Interpretable` impls
- No lifetimes at all when using `ParsePipelineText` or `ParseStatementText`

This is well-designed for the derive-heavy path.

## Concept Budget

### Use Case: "Add a new dialect with 3 operations"

| Concept | Where learned | Complexity |
|---------|--------------|------------|
| `Dialect` trait | kirin-ir | Low (just derive it) |
| `CompileTimeValue` | kirin-ir | Low (trait alias for Clone+Debug+Hash+PartialEq) |
| `Placeholder` | kirin-ir | Low (just impl `placeholder()`) |
| `#[kirin(type = T)]` | attribute | Low |
| `#[kirin(builders)]` | attribute | Low |
| `SSAValue` | kirin-ir | Medium (concept of SSA) |
| `ResultValue` | kirin-ir | Medium (output of an operation) |
| `Block` / `Region` | kirin-ir | Medium (when is each used?) |
| `Successor` | kirin-ir | Medium (for terminators) |
| `#[kirin(terminator)]` | attribute | Low |
| `#[kirin(pure)]` / `#[kirin(speculatable)]` | attribute | Low |
| `#[kirin(constant)]` | attribute | Low |
| `PhantomData<T>` pattern | Rust | Low but annoying |
| `#[wraps]` composition | attribute | Medium |

**Total: ~13 concepts.** This is manageable, especially since the first 5-6 are one-time learning.

## Toy Scenario Results

### Scenario: Define a TweezerPulse dialect

Mentally tracing through defining a pulse control dialect:
```rust
use kirin::prelude::*;

#[derive(Clone, Debug, Hash, PartialEq)]
pub enum PulseType { Duration, Amplitude, Phase }

impl Placeholder for PulseType {
    fn placeholder() -> Self { PulseType::Duration }
}

#[derive(Debug, Clone, PartialEq, Eq, Hash, Dialect, HasParser, PrettyPrint)]
#[kirin(builders, type = PulseType)]
pub enum TweezerPulse {
    #[chumsky(format = "{result:name} = {.ramp} {start}, {end}, {duration} -> {result:type}")]
    Ramp {
        start: SSAValue,
        end: SSAValue,
        duration: SSAValue,
        result: ResultValue,
    },
    // ... PhantomData needed!
    __Phantom(std::marker::PhantomData<PulseType>),
}
```

**What worked well:**
- `kirin::prelude::*` brings in most needed types
- `#[chumsky(format = "...")]` is declarative and readable
- `#[kirin(builders)]` auto-generates constructors
- The derive-heavy approach means minimal boilerplate for the operation definitions themselves

**What was confusing:**
- Had to remember `PhantomData` for the type parameter
- Need `Eq` on `PulseType` but it's not listed in `CompileTimeValue` bounds -- it's needed by `Dialect`'s `PartialEq` bound. The error message if you forget would be confusing.
- `Placeholder` vs `Default` distinction is not obvious from the trait alone

### Scenario: Find the right stage ID from a pipeline

Looking at `example/toy-lang/src/main.rs` lines 76-88, finding a stage by name is verbose:
```rust
let stage_id = pipeline.stages().iter().find_map(|s| {
    let sym = s.stage_name()?;
    let name = pipeline.resolve(sym)?;
    if name == stage_name { s.stage_id() } else { None }
}).ok_or_else(|| ...)?;
```
A convenience method `pipeline.stage_by_name("source")` would help.

### Scenario: Looking up and calling a function

Lines 91-110 of toy-lang's main.rs show 4 sequential lookups to go from function name to a callable specialization. This is deeply nested navigation: `pipeline -> function_by_name -> function_info -> staged_functions -> get(&stage_id) -> get_info(stage) -> specializations -> find active`. A convenience method would dramatically improve usability.

## Summary

- [P2] [confirmed] PhantomData boilerplate on every generic dialect type -- could be auto-generated by derive. `kirin-arith/src/lib.rs:128`, `kirin-scf/src/lib.rs:64,82,93`
- [P2] [confirmed] Long derive lists repeated on every type (8-10 derives). `example/toy-lang/src/language.rs:16-18`
- [P1] [confirmed] No convenience method to look up a stage by name from Pipeline. `example/toy-lang/src/main.rs:76-88`
- [P1] [confirmed] Function resolution from name to callable specialization is 6+ steps of nested lookups with no shorthand. `example/toy-lang/src/main.rs:91-174`
- [P3] [confirmed] `CompileTimeValue` is a trait alias; its relationship to `Eq` (required by `Dialect`) could confuse users who get errors about missing `Eq`.
- [P3] [likely] `#[wraps]` per-variant vs enum-level: two ways to do the same thing may confuse new users.
- [P2] [confirmed] `builders` attribute repeated on every dialect type could default to on.
