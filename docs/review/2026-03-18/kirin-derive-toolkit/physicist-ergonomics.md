# Physicist -- Ergonomics/DX Review: kirin-derive-toolkit

## Preamble

This crate is internal infrastructure. Users never import from it directly. The review focuses on the **user-visible output** of the derive macros it supports: the generated code quality, error messages, and the attribute surface area users must learn.

## Repetition & Boilerplate

### 1. Attribute namespace fragmentation
Users face multiple attribute namespaces across derives:
- `#[kirin(...)]` -- IR properties (type, terminator, pure, speculatable, constant, builders, default, into)
- `#[chumsky(...)]` -- Parser format strings
- `#[pretty(...)]` -- Pretty printer crate path
- `#[wraps]` -- Wrapper delegation
- `#[callable]` -- Interpreter callable marker
- `#[stage(...)]` -- Stage enum variant names
- `#[interpret(...)]` -- (removed, but was interpreter-specific)

A user defining a full-featured dialect must learn 5+ attribute namespaces. The separation is architecturally sound (each derive owns its namespace), but the cognitive load is real.

### 2. Attribute inconsistency: path vs string
- `#[kirin(crate = kirin_ir)]` uses path syntax
- `#[chumsky(format = "...")]` uses string syntax
- `#[stage(name = "source")]` uses string syntax

This is consistent within each namespace but inconsistent across them. Minor friction.

### 3. Generated builder naming convention
Builders generated by `#[kirin(builders)]` follow the pattern `Dialect::op_variant_name(stage, args...)`. For example:
- `Arith::<ArithType>::op_add(&mut stage, a, b)` returns a struct with `.result` and `.id`
- `Constant::<ArithValue, ArithType>::new(&mut stage, value)` (when `builders = new`)

The `op_` prefix and turbofish are verbose. Users writing programmatic IR construction (not parsing) must spell the full generic type. No way around this in Rust, but it's worth noting.

## Lifetime Complexity

No user-facing lifetime complexity from the toolkit itself. Generated code inserts lifetimes (`'__ir`, `'t`) but users never write these.

## Concept Budget

### Use Case: "Understand what the derive macros generate"

| Concept | Where learned | Complexity |
|---------|--------------|------------|
| `#[derive(Dialect)]` generates property traits | implicit | Low |
| `#[derive(Dialect)]` generates builder methods | implicit | Medium |
| `#[derive(HasParser)]` generates AST type + parser + EmitIR + ParseEmit | implicit | High (4 things) |
| `#[derive(PrettyPrint)]` generates printer | implicit | Low |
| `#[derive(Interpretable)]` generates match dispatch | implicit | Low |
| `#[derive(SSACFGRegion)]` generates entry_block dispatch | implicit | Low |
| `#[derive(StageMeta)]` generates stage metadata | implicit | Low |
| `#[derive(ParseDispatch)]` generates parser dispatch | implicit | Low |
| `#[derive(RenderDispatch)]` generates printer dispatch | implicit | Low |

**Total: ~9 derive macros.** The "which derives do I need?" question is the main confusion point.

A cheat sheet: "For a dialect: `Dialect + HasParser + PrettyPrint`. For interpreter: add `Interpretable + SSACFGRegion`. For a stage enum: `StageMeta + ParseDispatch + RenderDispatch`."

## Toy Scenario Results

### Scenario: Figure out which derives I need for a new dialect

Looking at toy-lang's `language.rs`:
```rust
#[derive(
    Debug, Clone, PartialEq, Eq, Hash, Dialect, HasParser, PrettyPrint, Interpretable, SSACFGRegion,
)]
```
That's 10 derives. A new user would not know to add `SSACFGRegion` unless they read the toy-lang example or got a compile error about missing `CallSemantics`. The relationship between `#[callable]` and `SSACFGRegion` is non-obvious.

### Scenario: Compile error from missing attribute

If I forget `#[kirin(type = T)]`, what happens? The derive should produce a clear error. If I mistype `#[chumsky(format = "...")]`, do I get a good diagnostic?

I cannot test this directly, but the use of `darling` for attribute parsing suggests errors would be reasonably descriptive (darling produces compile errors at the attribute site). This is a strength.

## Summary

- [P2] [confirmed] 5+ attribute namespaces across the derive ecosystem; a quick-reference guide mapping "what I want" to "which attribute" would help.
- [P2] [confirmed] 10 derives needed for a full-featured dialect with interpreter support. Consider a `#[derive(KirinDialect)]` shorthand that implies `Debug + Clone + PartialEq + Eq + Hash + Dialect + HasParser + PrettyPrint`.
- [P3] [confirmed] Relationship between `#[callable]` attribute and `#[derive(SSACFGRegion)]` is non-obvious; needs better documentation or a compile-time hint.
- [P3] [confirmed] Builder naming (`op_add`, `op_sub`) is discoverable via IDE but not documented in a central place.
